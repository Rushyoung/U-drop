# 后端架构与数据库设计

## 系统架构
- **logger**: 全链路异步审计日志 (Loguru)。
- **fastapi**: 业务路由中枢，支持 Phase 1-4。
- **service**: 业务逻辑层。包含滑动窗口聚合、配额预检、回收站流转、分享核销。
- **core**: 原生核心 (C-Core)、内存任务管理 (UploadsManager)、**WebSocket 调度器 (ConnectionManager)**。

## 数据库设计 (udrop.db)

### 1. users (用户表)
| 字段 | 类型 | 说明 |
| :--- | :--- | :--- |
| uuid | str | 唯一标识 (内部 PK) |
| storage_quota | bigint | 总配额 (Bytes) |
| used_storage | bigint | 已用量 (Bytes) |
| trash_expire_days | int | 回收站保留天数 |
| sync_seq | int | (待落盘) 当前同步序列号 |

### 2. devices (设备管理表)
| 字段 | 类型 | 说明 |
| :--- | :--- | :--- |
| device_id | str | PK |
| user_uuid | str | FK |
| device_name | str | 自定义名称 |
| last_seen | int | 最后活跃时间 |

## 实时信令系统 (WebSocket)
### 连接方式
`WS /api/v1/ws?token={bearer_token}`

### 信令协议 (JSON)
1. **INIT (服务端下发)**: 连接成功后的首条消息。
   `{"type": "INIT", "data": {"sync_seq": 10, "timestamp": 12345}}`
2. **MSG_NEW (广播)**: 有新消息或附件落位。
   `{"type": "MSG_NEW", "data": {"message_id": 105}, "sync_seq": 11}`
3. **MSG_DELETE (广播)**: 消息被删除。
   `{"type": "MSG_DELETE", "data": {"message_id": 105}, "sync_seq": 12}`
4. **UPLOAD_PROGRESS (广播)**: 跨端进度同步。
   `{"type": "UPLOAD_PROGRESS", "data": {"upload_id": "uuid", "received_size": 100, "total_size": 200}}`
5. **DEVICE_UPDATE (广播)**: 设备信息变更（如改名），用于刷新端侧缓存。
   `{"type": "DEVICE_UPDATE", "data": {"device_id": "uuid", "device_name": "新名字"}, "sync_seq": 13}`

## 核心业务规约
1. **多文件上传**: `POST /messages` 分配任务，`PATCH` 流式上传，`COMMIT` 后正式落位。
2. **回收站与配额**: 
   - 软删除不释放配额。
   - 物理删除（清空回收站）或 GC 清理时，**全额返还**逻辑配额并减少物理引用。
3. **逻辑寻址与 CAS**: 
   - 前端通过 `attachment_id` (逻辑 ID) 进行下载和缩略图获取，后端执行严格的所有权校验（IDOR 防御）。
   - 物理哈希 (`full_hash`) 仅用于底层内容寻址和秒传校验，对用户透明，实现了业务逻辑与物理存储的彻底解耦。
   - 该架构天然支持多存储后端（本地、WebDAV、S3 等），由 `file_service` 内部决定物理路由。
4. **同步序列号 (Sync Seq)**: 每次时间线变动（增/删/复活），用户 Seq 加 1，用于端侧判断数据是否需要增量对齐。
