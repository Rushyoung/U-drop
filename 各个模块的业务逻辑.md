# 各个模块的业务逻辑 (更新版)

## 1. 消息与文件流转逻辑 (Telegram 风格)

现在我们采用“先占位，后补全”的策略，这样用户发文件时体验最快。

1.  **消息创建阶段**：
    *   用户想发文件？先调 `POST /messages`。
    *   传个文件名、大概大小和快速哈希（Sparse MD5）。
    *   服务端直接在数据库里塞一个消息，`file_hash` 先空着。
    *   同时在内存里开个“上传任务”（UploadTask），给客户端发个 `upload_id`。
    *   这时候别的设备拉时间线，已经能看到这根消息了，只是附件显示“上传中”。

2.  **上传劫持阶段**：
    *   客户端往 `PATCH /files/upload/{upload_id}` 丢数据流。
    *   **关键点**：客户端一边传，一边在本地算全量哈希（Blake3）。
    *   一旦算出来了，立刻在 Header 里带上 `X-Full-Hash`。
    *   服务端一看到这个 Header，立马查库。要是库里已经有这个文件了，服务端直接“咔嚓”断开连接，告诉客户端：“别传了，我这儿有，已经帮你接好了。”

3.  **最终校验阶段**：
    *   如果没被劫持，流传完了，客户端调 `POST /commit`。
    *   服务端自己再算一遍 Blake3，跟客户端传过来的比。
    *   **不一致就报错**，直接删掉临时文件，不入库，保证数据绝对干净。
    *   一致的话，把临时文件挪到正式仓库，把哈希填回消息表，收工。

## 2. 存储与清理逻辑

*   **引用计数**：每个文件在 `file_info` 里都有个 `refer_count`。
*   **删除消息**：你删一条消息，它引用的文件计数就减1。
*   **物理清理**：服务端会定期扫一遍，看到哪个文件计数变成 0 了，就说明没人要了，这时候再物理删除文件，省得硬盘爆掉。

## 3. 任务管理

*   **内存管理器**：为了快，上传任务不写数据库了，直接丢在内存里的 `dict`。
*   **重启后果**：如果服务器重启，正在传的任务就没了，客户端需要重新从 `POST /messages` 开始，反正有秒传，也不费事。
