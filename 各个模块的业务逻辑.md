# 各个模块的业务逻辑设计

## 1. 用户认证与设备管理 (Auth & Device MGMT)
### 1.1 注册与登录
- **注册**: 校验账号唯一性，对密码加盐存储，分配 UUID。
- **登录**: 验证凭据，生成 Bearer Token (Session)，记录登录设备的 `device_id`。
- **设备识别**: 登录时由客户端上传 `device_id` (如安卓 UUID 或浏览器指纹) 和 `device_name`。
- **多设备管理**: 用户可查询 `devices` 表查看已登录设备，通过接口强制注销 (Logout)，后端通过 WebSocket 发送 `logout` 控制指令。

## 2. 时间线与消息管理 (Timeline & Messaging)
### 2.1 消息发送
- **发送逻辑**: 支持文本消息、图片和文件引用。
- **Hashtag 处理**: 提取文本中的 `#` 标签，写入 `hashtags` 表，并通过 `messages_tags` 建立关联，便于搜索。
- **关联文件**: 若包含文件，需关联 `file_info` 中的哈希值。
### 2.2 搜索功能
- **全文搜索**: 对 `content` 字段进行模糊匹配。
- **标签搜索**: 按标签名检索，并通过 `messages_tags` 关联到消息。
- **时间线刷新**: 按“当前消息的 id”分段拉取：首次拉最近一屏；向上翻页拉更早消息；收到刷新信号后拉更新消息。
### 2.3 消息删除
- **逻辑**: 设置 `is_deleted = 1`，查询时过滤。删除后服务端发送 `REFRESH_TIMELINE`，客户端调用同步接口拉取删除消息 id 并更新本地列表。如果是最后一条引用该文件的消息，触发文件清理机制（引用计数）。

## 3. 文件管理与闪传 (File & Flash Transfer)
### 3.1 阶段式哈希匹配 (闪传)
- **阶段 1**: 客户端上传 Sparse MD5 (头、中、尾各 4KB)。
- **阶段 2**: 后端匹配 `file_info.sparse_hash`。
  - 若**未匹配**: 进行常规上传流程。
  - 若**已匹配**: 客户端继续上传全量 Blake3 哈希。
- **阶段 3**: 后端验证 Blake3。
  - 若一致: 实现闪传 (增加 `refer_count`)。
  - 若冲突: 继续上传文件块。
### 3.2 文件预览图
- **生成**: 后端接收图片文件后，异步调用 C-Core (.dll/.so) 的图像处理函数 (stb_image) 生成缩略图。
- **下发**: 将生成的缩略图路径存入 `thumbnail_url`，供客户端在展示时间线时使用。
### 3.3 下载逻辑
- **重命名**: 使用 FastAPI 的 `FileResponse` 或 `StreamingResponse`，设置 `Content-Disposition` 头部以支持原始文件名重命名。

## 4. WebSocket 信令控制 (WS Signaling)
### 4.1 心跳机制
- 客户端与后端建立连接，每 30s 发送一次心跳维持。
- 服务端记录 `active_sessions`。
### 4.2 控制信息下发
- **时间线更新**: 其它设备发送消息后，后端向所有在线设备推送 `REFRESH_TIMELINE`。
- **远程下线**: 撤销 Token 后，向特定设备的 WebSocket 发送 `LOGOUT` 指令。

## 5. 原生核心集成 (Native Core Engine)
### 5.1 共享逻辑
- **哈希计算**: 统一使用 C 编写的 Blake3 和 Sparse MD5 算法。
- **图像缩放**: 使用 `stb_image_resize` 实现高性能缩略图生成。
### 5.2 跨平台适配
- **安卓**: 通过 JNI 调用 `.so` 库。
- **网页**: 将 C 代码编译为 WASM，供浏览器端进行哈希计算。
- **后端**: Python 通过 `ctypes` 调用 `.dll` 或 `.so`。
